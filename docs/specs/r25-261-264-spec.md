### Relevant findings by issue

#### #261 (secure writes / TOCTOU)
1. **`scripts/derive_keys.py` uses non-atomic direct writes**:
   - `open(..., "wb")` then `os.chmod(..., 0o600)` for private key.
   - This creates a TOCTOU exposure window where permissions depend on umask before chmod.
2. Existing `atomic_write()` helper is good for atomic replacement + restrictive perms, but several command paths rely on pre-checks like `if exists and not --overwrite` before writing. This is logically race-prone for strict no-overwrite semantics.

#### #262 (rotation signature binds only sign key)
- `scripts/rotate_keys.py` currently computes:
  - `rotation_signature = old_private_key.sign(new_sign_pub_bytes)`
- Risk: `new_public_enc_key` is not cryptographically bound by signature. A tampered payload could swap `new_public_enc_key` while keeping the signature valid for `new_public_sign_key`.

#### #263 (JWT stdout default leak)
- `scripts/authenticate.py` prints JWT to stdout by default when `--save-token` is not provided.
- This leaks secrets into shell transcripts, CI logs, command capture tooling, and accidental copy/paste.

#### #264 (pinned deps / docs)
- `requirements.txt` and `requirements-dev.txt` are range-based (`>=,<`).
- Script headers and `SKILL.md` still show ad-hoc install guidance (`pip install cryptography requests`) instead of pinned workflow.

## Issue #261 — Secure file writes / TOCTOU

### Security rationale
- Direct write + later chmod can briefly expose secrets with weaker permissions.
- Pre-check (`exists`) + later write is a classic race for no-overwrite semantics.

### Implementation spec

#### 261-A: eliminate direct private-key writes in `derive_keys.py`
- Replace direct writes of private artifacts with helper-based atomic writes:
  - SSH private key (`id_agent_ed25519`) -> `atomic_write(..., mode=0o600)`
  - PGP private key (`agent_pgp_private.asc`) -> `atomic_write(..., mode=0o600)`
- Public files may remain mode `0o644` (or default) but should still be atomic for consistency.

#### 261-B: add strict no-overwrite helper path
- In `scripts/crypto_utils.py`, add a helper for no-overwrite writes with atomic semantics for create-only behavior:
  - Suggested name: `atomic_write_new(path, content, mode=0o600)`
  - Must fail if destination already exists (atomic create semantics, no pre-check race).
- Use this helper in code paths that currently do `exists` pre-check for user-provided outputs where `--overwrite` is false:
  - `scripts/keygen.py`
  - `scripts/register.py`
  - `scripts/rotate_keys.py` (manual mode outputs `--new-keys`, `--payload`)
- Keep current `atomic_write()` for replacement semantics where overwrite is intended.

### Backward compatibility
- No CLI flag changes required.
- Behavior remains functionally same for normal users.
- In race edge-cases, operation now fails safely instead of clobbering existing files.

### TDD plan (write tests first)
1. **New unit test**: `atomic_write_new` fails when target exists.
2. **New integration tests**:
   - keygen/register manual writes with no `--overwrite` do not replace existing files.
3. **Refactor coverage test**:
   - derive_keys private outputs created with mode `0600`.
4. Ensure existing tests still pass.

### Acceptance criteria
- [ ] No direct secret-file writes remain in `derive_keys.py`.
- [ ] `derive_keys.py` private output files are written atomically and mode `0600`.
- [ ] `atomic_write_new` exists and enforces create-only (fails if path exists).
- [ ] `keygen.py`, `register.py`, `rotate_keys.py` use atomic create-only path when `--overwrite` is false.
- [ ] Tests demonstrate no-overwrite race-safe intent and pass in CI.

## Issue #262 — Rotation signature must bind encryption key

### Security rationale
- Current signature proves ownership over new signing key only.
- Attack class: payload tampering that substitutes `new_public_enc_key` without invalidating signature.

### Implementation spec

#### 262-A: canonical signed message including both keys
- In `scripts/rotate_keys.py`, define canonical bytes for rotation signature:
  - `context || new_sign_pub_bytes || new_enc_pub_bytes`
  - `context` constant (domain separator), e.g. `b"agent-id:rotate:v2:"`
- Generate signature over canonical bytes (not just signing key bytes).
- Keep payload fields unchanged:
  - `new_public_sign_key`
  - `new_public_enc_key`
  - `rotation_signature`

#### 262-B: server compatibility/migration strategy
- Since API reference currently states old semantics, migration should be explicit:
  - Preferred: backend accepts both during rollout (`v1=sign-only`, `v2=bound`) and then deprecates v1.
  - Client default should emit v2 signature after rollout window.
- Optional client fallback (if required by rollout):
  - `--legacy-rotation-signature` temporary flag (deprecated), default off.

### Backward compatibility
- Potential protocol break if backend verifies old message only.
- Mitigation: rollout gate or temporary compatibility mode.

### TDD plan
1. **Unit test**: signature message builder deterministic and includes both key bytes.
2. **Unit test**: changing only `new_public_enc_key` invalidates old signature verification.
3. **Integration test**: rotation payload generation produces v2 signature and still preserves metadata (`agent_id`, `display_name`).
4. (If fallback flag used) test both code paths and default selection.

### Acceptance criteria
- [ ] Rotation signature input binds both `new_public_sign_key` and `new_public_enc_key`.
- [ ] Tests prove enc-key substitution causes verification failure.
- [ ] Payload schema remains unchanged (no breaking JSON field rename).
- [ ] Migration strategy documented (dual-accept window or explicit compatibility flag).

## Issue #263 — JWT stdout leak default

### Security rationale
- Default stdout token emission is unsafe-by-default in automated environments.
- Principle: secrets should not be emitted unless explicitly requested.

### Implementation spec

#### 263-A: explicit output intent for token
- `scripts/authenticate.py` CLI behavior:
  - Add `--print-token` flag to explicitly allow stdout token output.
  - Keep `--save-token <path>` for secure file output (0600 via existing helper).
  - If neither is supplied: **fail closed** with clear error message and non-zero exit.
- Keep expiry metadata on stderr.

#### 263-B: docs and examples update
- Update `SKILL.md` examples to use:
  - `--save-token` (recommended)
  - `--print-token` only for explicit piping/advanced usage

### Backward compatibility
- **Intentional breaking change**: previous default stdout token no longer works.
- Migration is straightforward and explicit.

### CLI migration notes (required)

Old behavior:
```bash
TOKEN=$(python3 scripts/authenticate.py agent_keys.json)
```

New recommended behavior:
```bash
python3 scripts/authenticate.py agent_keys.json --save-token /tmp/agent_token.jwt
TOKEN=$(cat /tmp/agent_token.jwt)
```

New explicit stdout behavior (opt-in):
```bash
TOKEN=$(python3 scripts/authenticate.py agent_keys.json --print-token)
```

### TDD plan
1. **New test**: authenticate without `--save-token` and without `--print-token` exits non-zero and does not emit token.
2. **New test**: `--print-token` prints token to stdout only.
3. **Existing test adjust/retain**: `--save-token` writes 0600 file and prints only status to stderr.
4. Negative test: ensure token not present in stderr in default failure path.

### Acceptance criteria
- [ ] Default authenticate flow no longer prints JWT to stdout.
- [ ] `--print-token` restores explicit stdout behavior.
- [ ] `--save-token` remains secure (mode `0600`) and documented as recommended.
- [ ] Docs and examples reflect new safe default and migration snippet.

## Issue #264 — Pinned deps + doc guidance

### Security rationale
- Range pins can drift silently and alter runtime/security posture.
- Reproducible environments require exact pins and a documented update flow.

### Implementation spec

#### 264-A: pin direct dependencies exactly
- Update `requirements.txt` to exact versions (`==`) for runtime deps.
- Update `requirements-dev.txt` to exact versions (`==`) for dev/test tooling.
- Keep dependency set minimal (no unnecessary additions).

#### 264-B: documentation hardening
- In `SKILL.md` and script headers:
  - Replace ad-hoc `pip install ...` with `pip install -r requirements.txt` (or `requirements-dev.txt` for tests/dev).
- Add a short dependency policy section in docs:
  - how to bump pins
  - run `pip-audit`
  - run test suite before merge

### Backward compatibility
- No runtime API/CLI break.
- Dev/install workflows become deterministic.

### TDD/verification plan
1. Add a lightweight test or CI assertion that requirement lines are exact pins for direct deps.
2. Ensure CI still installs and runs tests/lint/security successfully with pinned files.

### Acceptance criteria
- [ ] `requirements.txt` and `requirements-dev.txt` use exact pins (`==`) for direct dependencies.
- [ ] `SKILL.md` install instructions point to requirements files.
- [ ] Script-level “Requires” guidance aligns with pinned requirements workflow.
- [ ] CI jobs pass using pinned dependency files.
